The Front Lawn

Where every visitor arrives.

src/app/page.tsx
›

The public homepage — the first thing a client sees. It's a Server Component: it runs on the server, renders a full-viewport hero, queries the database for a “Selected Work” grid, and ends with a booking form. Nothing ships to your phone until it's finished HTML.

What lives here
  • src/app/page.tsxThe homepage. Queries PortfolioGroups that have photos + a cover image.
  • src/app/layout.tsxRoot layout — fonts and chrome wrapping every page.
  • mobile-menu.tsx / reveal.tsxClient components: nav menu + scroll-reveal animations.
  • src/app/booking-form.tsxThe contact form that feeds the Booking table.
How it flows
1
A visitor requests /; Next.js runs Home() on the server.
2
It calls auth() (to show an Admin link if logged in) and prisma.portfolioGroup.findMany().
3
Only groups with some portfolios AND a coverImageUrl make the grid — empties are hidden.
4
Each card links to /portfolio/[slug]. Finished HTML ships to the phone.
Quest

✓ The query filters on portfolios:{some:{}} and coverImageUrl:{not:null}.

The grid is public; login only toggles the Admin link.

The Gallery Vault

Password-protected client deliveries.

src/app/gallery/[slug]
›

Separate from the public portfolio: this is how finished photos are delivered privately to a client. Each Gallery has a random password. The client opens a link, types it once, and a signed cookie remembers them for 30 days. Inside: a lightbox, favorites selection, and a ZIP download.

What lives here
  • gallery/[slug]/page.tsxLoads the gallery, checks the access cookie, gates on the password form.
  • password-form.tsxClient form posting to verifyAlbumPassword.
  • gallery-client.tsx / lightbox.tsxThe interactive viewer + selection UI.
  • api/gallery/[slug]/download/route.tsStreams selected photos as a ZIP (archiver).
How it flows
1
Client visits /gallery/emma-wedding; page reads cookie gallery-emma-wedding-access.
2
No cookie → render the password form instead of photos.
3
verifyAlbumPassword() compares against gallery.password.
4
Match → set an httpOnly cookie for 30 days, then revalidate.
5
Cookie now exists → the full lightbox + ZIP download renders.
Quest
How does the gallery remember a client already entered the password?
✓ verifyAlbumPassword sets gallery-<slug>-access=granted, maxAge 30 days.
No IP logic — it's a cookie.
Clients are not NextAuth users; only admins log in.

The Darkroom

Where uploaded photos are developed.

src/app/api/uploadthing/core.ts
›

The upload pipeline. Photos never touch the server's disk — they go to UploadThing. Three routers (gallery photos, portfolio photos, group covers) each verify you're admin, record the photo in a transaction at the next sortOrder, revalidate the right pages, then — without blocking — backfill width & height with sharp.

What lives here
  • api/uploadthing/core.tsThe FileRouter: galleryPhoto, portfolioPhoto, groupCoverImage.
  • api/uploadthing/route.tsWires the router into a Next.js route handler.
  • lib/uploadthing.ts + hooks/use-upload-queue.tsClient helpers + a resilient upload queue (tested).
How it flows
1
.middleware() runs — rejects anyone whose role isn't ADMIN.
2
File lands in cloud storage; .onUploadComplete() fires on the server.
3
A $transaction reads max(sortOrder) and inserts the photo at max+1.
4
revalidatePath() refreshes both the public and admin pages.
5
Then import('sharp') lazily reads dimensions and updates the row — fire-and-forget.
Quest
Why read width/height with sharp in a fire-and-forget block?
✓ The comment says exactly this; the import('sharp').then(...) chain isn't awaited.
Not the reason — it's about response latency.
The whole route already required admin in the middleware.

The Gatehouse

Identity, sessions & route guards.

auth.ts · dal.ts · proxy.ts
›

Three cooperating guards. auth.ts configures NextAuth (email + bcrypt password, JWT sessions). proxy.ts is edge middleware that redirects at navigation time. dal.ts holds verifyAdmin() — the last word before protected data loads.

What lives here
  • src/auth.tsNextAuth v5: Credentials provider, bcrypt compare, JWT callbacks with id + role.
  • src/lib/dal.tsverifyAdmin() and getCurrentUser(), both React-cached.
  • src/proxy.tsMiddleware: bounce guests off /admin, logged-in users off /login.
How it flows
1
Login posts email + password; authorize() looks up the user.
2
bcrypt.compare() validates; failure returns null → rejected.
3
On success a JWT is minted; the jwt callback adds id and role.
4
The session callback copies them onto session.user.
Quest
Where do id and role come from when a page reads session.user.role?
✓ Session strategy is jwt; no per-page DB hit for the role.
They ride in the token instead.
The server mints the JWT; the browser can't set role.

The Action Bench

Every write to the world.

src/app/actions/*
›

Server Actions are the mutation layer — the sanctioned way to change data. The pattern is consistent: verifyAdmin → validate with Zod → touch Prisma → revalidatePath → redirect. Public actions (booking) swap the admin check for a CAPTCHA.

What lives here
  • actions/gallery.tscreate/update/delete, regenerate password, verifyAlbumPassword.
  • actions/portfolio*.tsPublic content CRUD + reordering. Tested.
  • actions/booking.tsPublic: validates + Turnstile CAPTCHA, then stores a Booking.
How it flows
1
A form invokes a 'use server' action.
2
Admin actions await verifyAdmin() first — no exceptions.
3
Input is parsed with a zod schema; failures return typed errors.
4
Prisma writes; slugs are made with a local slugify().
5
revalidatePath() refreshes pages; many end in redirect().
Quest
A brand-new gallery gets its password from…
✓ createGallery calls generatePassword(); admins regenerate later.
It's generated automatically on create.
Slug and password are independent.

The Booking Gate

Turning visitors into clients.

booking-form.tsx · turnstile.ts
›

The homepage contact form. To keep spam out, it's guarded by Cloudflare Turnstile — a privacy-friendly CAPTCHA. The widget sits below the submit button; on submit, the server re-verifies the token with Cloudflare before a Booking is written.

What lives here
  • src/app/booking-form.tsxClient form + Turnstile widget (@marsidev/react-turnstile).
  • src/lib/turnstile.tsServer-side token verification against Cloudflare.
  • actions/booking.tsValidates fields, verifies Turnstile, creates the Booking.
How it flows
1
Visitor fills the form and solves Turnstile → gets a token.
2
Form submits the token with the fields to the action.
3
Server verifies the token with Cloudflare's siteverify.
4
Valid → a Booking row is created with status PENDING.
Quest
The Turnstile token from the browser is trusted because…
✓ Client widgets can be bypassed, so lib/turnstile.ts does a server check.
Never trust the client alone — hence the server verify.
It's verified immediately, per submission.

Deliver photos to a client

Upload → private gallery → download

›
🎛️

1 · Admin creates a gallery

actions/gallery.ts → createGallery

Faith names a gallery. The action slugifies the title and generates a random password.

const slug = slugify(title) await prisma.gallery.create({ data: { title, slug, password: generatePassword() // 8 hex chars }})
🧪

2 · She uploads photos

api/uploadthing/core.ts → galleryPhoto

Middleware confirms she's ADMIN. Files go to UploadThing. On complete, a transaction inserts each photo at max(sortOrder)+1.

await prisma.$transaction(async (tx) => { const m = await tx.photo.aggregate({ _max: { sortOrder: true } }) return tx.photo.create({ data: { url: file.ufsUrl, fileKey: file.key, sortOrder: (m._max.sortOrder ?? -1) + 1 }}) })
📐

3 · Dimensions fill in quietly

core.ts (fire-and-forget)

Without blocking the response, sharp is lazily imported, the image fetched, and width/height written back.

import('sharp').then(({default: sharp}) => fetch(file.ufsUrl) .then(r => r.arrayBuffer()) .then(buf => sharp(Buffer.from(buf)).metadata()) .then(meta => prisma.photo.update(...)))
🔗

4 · Client opens the link

gallery/[slug]/page.tsx

The page reads the per-gallery cookie. Missing → it renders only the password form.

const c = cookieStore.get(`gallery-${slug}-access`) if (c?.value !== 'granted') return <AlbumPasswordForm slug={slug} />
🔐

5 · Password unlocks it

actions/gallery.ts → verifyAlbumPassword

On a correct password, a 30-day httpOnly cookie is set and the page revalidates — the lightbox renders.

if (gallery.password === password) cookieStore.set(`gallery-${slug}-access`, 'granted', { httpOnly: true, maxAge: 60*60*24*30 })
⬇️

6 · They download favorites

api/gallery/[slug]/download/route.ts

Selected photos are streamed as a ZIP built on the fly with archiver — no temp files.

// archiver streams selected photos const archive = archiver('zip') photos.forEach(p => archive.append(stream, ...)) archive.finalize()
🔑

Admin logs in & edits

Credentials → JWT → guarded page

›
🚦

1 · Hitting /admin logged out

src/proxy.ts (middleware)

Edge middleware runs first. No session on an /admin path → immediate redirect to /login.

if (pathname.startsWith('/admin') && !session?.user) return NextResponse.redirect( new URL('/login', request.url))
🛡️

2 · Submitting credentials

src/auth.ts → authorize()

NextAuth looks up the user by email and bcrypt-compares the password. A mismatch fails login.

const user = await prisma.user.findUnique({ where: { email } }) const ok = await bcrypt.compare( password, user.password) if (!ok) return null
🎫

3 · A token is minted

src/auth.ts → callbacks

Session strategy is JWT. The jwt callback writes id + role; the session callback copies them onto session.user.

async jwt({ token, user }) { if (user) { token.id = user.id; token.role = user.role } return token }
🎛️

4 · The admin page loads

src/lib/dal.ts → verifyAdmin()

Inside the page, verifyAdmin() re-checks role === 'ADMIN'. It's React-cached so repeats are free.

export const verifyAdmin = cache(async () => { const s = await auth() if (s?.user?.role !== 'ADMIN') redirect('/login') return s })
⚙️

5 · An edit is saved

app/actions/*.ts

The form invokes a server action that ALSO starts with verifyAdmin(), validates with zod, writes, then revalidates.

'use server' await verifyAdmin() const parsed = schema.safeParse(input) await prisma.gallery.update(...) revalidatePath(`/admin/galleries/${id}`)
✉️
