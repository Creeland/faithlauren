gallery-access

The access scheme becomes a secret.

src/lib/gallery-access.ts
3 sites → 1 owner

The page, the ZIP download route, and the password action each hand-rolled the cookie: same name template, same magic string, same 30-day math. Any change — a longer lifetime, a signed value — meant a three-file scavenger hunt, and missing one meant clients locked out of downloads but not pages (or worse, the reverse).

Before — in three different files
// gallery/[slug]/page.tsx const c = cookieStore.get(`gallery-${slug}-access`) const hasAccess = c?.value === "granted" // api/gallery/[slug]/download/route.ts const c = cookieStore.get(`gallery-${slug}-access`) if (c?.value !== "granted") return 401 // actions/gallery.ts cookieStore.set(`gallery-${slug}-access`, "granted", { httpOnly: true, maxAge: 60*60*24*30, ... })
After — one module, two questions
// lib/gallery-access.ts owns the whole scheme await grantGalleryAccess(slug) // action await hasGalleryAccess(slug) // page & route
Now hidden inside
  • The cookie name format — no caller can typo the template string anymore.
  • The "granted" sentinel & 30-day lifetime — change access duration in exactly one line.
  • httpOnly / secure / sameSite flags — the security posture is one decision, not three.

sortable

One way to persist a drag-and-drop.

src/lib/sortable.ts
4 copies + 2 fixes

Four reorder actions (photos, portfolio photos, portfolios, groups) each parsed JSON off the form, cast it without validation, and looped updates. Worse: two used $transaction, two used Promise.all — so a failure mid-reorder of groups could commit half the new order. That's not a style nit; it's a data-integrity difference nobody chose on purpose.

Before — ×4, two of them non-atomic
const order = JSON.parse(formData.get("order")) as { id: string; sortOrder: number }[] // trust the client! await Promise.all( // ⚠ partial failure = order.map((item) => // half-reordered list prisma.portfolio.update({ ... })))
After — validated, atomic, everywhere
await persistOrder(prisma.portfolio, formData) // inside: zod-validated payload, then // prisma.$transaction — all positions or none
Now hidden inside
  • Payload validation — a zod schema rejects malformed order arrays before any write.
  • Atomicity policy — every reorder in the app is transactional, by construction.
  • The wire format — that ordering travels as a JSON string in a form field is now nobody's business but this module's.

storage

Deleting cloud files stops being a ritual.

src/lib/storage.ts
8 sites, 5 clients → 1

Five files each constructed their own new UTApi() and repeated the same incantation: map to fileKeys, filter out nulls, cast, check non-empty, then delete. Eight times. Every future caller had to remember all four steps or leak files / crash on nulls.

Before — ×8 across five files
const utapi = new UTApi() // per file! const fileKeys = photos.map((p) => p.fileKey) .filter(Boolean) as string[] if (fileKeys.length > 0) { await utapi.deleteFiles(fileKeys) }
After — pass the raw rows
await deleteStoredFiles(photos.map((p) => p.fileKey)) // nulls skipped, empty = no-op, one shared client
Now hidden inside
  • Null handling — the type accepts (string | null | undefined)[], so callers pass DB values as-is.
  • The empty no-op — "don't call the API with nothing" is defined out of existence for callers.
  • The UTApi client — one instance, one place to add retries or logging later.

photo-store

How a photo lands is one decision.

src/lib/photo-store.ts
core.ts −40%

Both upload routers duplicated the trickiest code in the app: a transaction that claims max(sortOrder)+1 (so concurrent uploads can't collide) and the unawaited sharp pipeline that backfills image dimensions. Subtle concurrency policy, written twice, drifting independently. The routers now do only router things: auth, existence checks, revalidation.

Before — ×2 inside core.ts
const photo = await prisma.$transaction(async (tx) => { const max = await tx.photo.aggregate({ _max: { sortOrder: true } }) return tx.photo.create({ data: { ..., sortOrder: (max._max.sortOrder ?? -1) + 1 } }) }) // + 12 more lines of sharp backfill … twice
After — the router reads like a sentence
const photo = await addGalleryPhoto(galleryId, { url: file.ufsUrl, key: file.key, name: file.name, })
Now hidden inside
  • The append position policy — transactionally claimed, safe under concurrent uploads.
  • The sharp backfill — lazy import, fetch, measure, update, swallow errors, never block the response.
  • What stayed OUT on purpose — revalidatePath calls remain in core.ts: which URLs refresh is a routing concern, not a storage concern.

dal · requireAdmin

One check, two failure modes.

src/lib/dal.ts
3 checks → 1 (−2 any)

Every upload middleware re-implemented the admin check inline — with an (session.user as any).role cast that wasn't even needed, since types/next-auth.d.ts already types the role. The existing verifyAdmin() couldn't be reused because it redirects, which is meaningless mid-upload. The insight: it's the same check with a context-appropriate failure mode.

Before — ×3 in core.ts
const session = await auth() if (!session?.user || (session.user as any).role !== "ADMIN") { throw new Error("Unauthorized") }
After — siblings in the auth boundary
await verifyAdmin() // pages: redirect to /login await requireAdmin() // APIs: throw Unauthorized
Now hidden inside
  • The role predicate — what "is an admin" means lives only in dal.ts.
  • React cache() — repeated checks within one request are free, in both variants.
  • Typed sessions — the as any casts are gone; the type augmentation does its job.

slugify

One URL grammar for the whole site.

src/lib/slugify.ts
3 copies → 1

The smallest change but the clearest lesson: the exact same six lines defined "what a slug is" in three files. Three copies of a design decision aren't one decision — they're three that happen to agree today. Fix a slug bug in gallery.ts and portfolios silently keep the old behavior; URLs and uniqueness checks drift apart.

Before — ×3, character for character
// gallery.ts, portfolio.ts, portfolio-group.ts function slugify(text: string) { return text.toLowerCase() .replace(/[^a-z0-9]+/g, "-") .replace(/(^-|-$)+/g, "") }
After — one definition, now tested
import { slugify } from "@/lib/slugify" // slugify.test.ts pins the grammar down
Now guaranteed
  • Every content type agrees on URL shape, forever, by construction.
  • The grammar has tests — unicode dashes, punctuation runs, and edge cases are pinned.

Step 3 · Why It Works

The philosophy under the diff.

Four ideas from Ousterhout's A Philosophy of Software Design, each mapped to the exact place this refactor used it.

WIDE INTERFACE
thin logic
Shallow
costs almost as much to learn as it saves
small
lots of functionality hidden below
Deep
one call buys a whole policy

📦 Modules should be deep

The interface is the cost every caller pays forever; the implementation is the benefit. Maximize the ratio.

Applied: grantGalleryAccess(slug) — one string in, an entire authentication scheme out. The caller's knowledge burden dropped from five cookie details to one verb.

🙈 Information hiding

A module should encapsulate design decisions — formats, algorithms, policies — so callers physically cannot depend on them.

Applied: nobody outside sortable.ts knows ordering travels as JSON in a form field; nobody outside storage.ts knows UploadThing hates empty delete calls. Those facts can now change without a repo-wide grep.

🧯 Define errors out of existence

The best error handling is an interface where the error can't occur. Handle edge cases inside, don't export them.

Applied: deleteStoredFiles accepts nulls and empty lists — the "remember to filter and check length" failure mode no longer exists for callers. persistOrder rejects malformed payloads before any write can half-apply.

⬇️ Pull complexity downward

It's better for a module to be internally complex than to push complexity onto its many callers.

Applied: photo-store.ts is the most intricate file in lib/ — transactions, races, unawaited pipelines. That's the point: two upload routers got simple because one module agreed to be complicated.

⚖️ …and knowing when to stop

Depth is a ratio, not a religion. A wrapper that hides nothing is just another interface to learn.

Applied: revalidatePath calls stayed in the routers (routing concern, not storage). The create-actions' inline aggregate for top-level sortOrder stayed put — three readable lines each, different transactional needs; a shared helper would have added interface without hiding a decision. And generatePassword() stayed private to gallery.ts — it has exactly one caller.
